## Executive Summary

The branch is functionally strong for core CRM operations. The Docker stack starts, Prisma is current, backend tests pass, frontend builds cleanly, and the Playwright smoke suite proves the main Chromium happy paths: auth, customers, projects, quotes, contracts, calendar, admin, reports entry, and Project 360 navigation. Documents v1 is materially improved: render/download semantics are sane, quote and contract PDFs work, and template runtime gating is present for `QUOTATION` and `CONTRACT`.

The main blockers are not feature count but production posture and consistency. The current branch still exposes too much settings data publicly, stores auth tokens in JavaScript-readable browser storage, and applies permission-based RBAC unevenly across core CRM endpoints. Documentation also overstates maturity in several places, and automated test breadth is still far below an 80% confidence target. For an internal staging/demo environment this branch is credible. For internet-exposed production, it is not yet ship-ready without a short, focused hardening pass.

## Confirmed Defects

### 1. Critical — Public settings endpoints leak internal company and policy data

- Subsystem: Security / Backend API
- Evidence:
  - `backend/src/settings/settings.controller.ts:15-99`
  - `backend/src/settings/settings.service.ts:30-41`
  - `backend/src/settings/settings.service.ts:97-109`
  - `backend/src/settings/settings.service.ts:130-143`
  - Runtime check: unauthenticated `GET /api/settings` returned `200`
  - Runtime check: unauthenticated `GET /api/settings/policies` returned `200`
- What is exposed:
  - company tax ID, bank account, bank account name, bank branch, address, email, phone
  - policy text from settings
  - logo payload through the settings surface
- Impact:
  - Data disclosure to any anonymous caller
  - Violates least privilege and data minimization
  - Public login/topbar branding needs only a narrow subset, not full operational settings
- Recommendation:
  - Put `@UseGuards(JwtAuthGuard)` at controller level for settings
  - Keep only `company` and `logo` public if needed
  - Split public company DTO from internal company settings DTO
  - Remove bank/policy data from all public responses
- Effort: 2-4h

### 2. High — Access and refresh tokens are stored in JavaScript-readable cookies and localStorage

- Subsystem: Security / Frontend auth
- Evidence:
  - `frontend/lib/auth.ts:11-17`
  - `frontend/lib/auth.ts:24-35`
  - `frontend/lib/auth.ts:123-141`
  - `frontend/hooks/use-auth.ts:52-59`
- Details:
  - access token and refresh token are written via `document.cookie`
  - cookies are `SameSite=Lax` only, with no `Secure`
  - user object persists in `localStorage`
  - tokens are later read back from JavaScript
- Impact:
  - Any successful XSS can steal both access and refresh tokens
  - Refresh token is not isolated in an `HttpOnly` cookie
  - Browser storage compromise means full session compromise
- Recommendation:
  - Move refresh token to backend-set `HttpOnly; Secure; SameSite=Lax/Strict` cookie
  - Keep access token in memory or shorten its lifetime further
  - Remove refresh token from JS-visible storage
- Effort: 8-16h

### 3. High — Permission-based RBAC is inconsistent across core CRM controllers

- Subsystem: Security / Authorization / Architecture
- Evidence:
  - `backend/src/customers/customers.controller.ts:12-60`
  - `backend/src/projects/projects.controller.ts:14-99`
  - `backend/src/quotes/quotes.controller.ts:18-94`
  - `backend/src/contracts/contracts.controller.ts:16-92`
  - `backend/src/activities/activities.controller.ts:11-42`
  - Compare with guarded admin surfaces:
    - `backend/src/settings/settings.controller.ts:42-49`
    - `backend/src/roles/roles.controller.ts:45-79`
    - `backend/src/common/guards/permissions.guard.ts:23-50`
- Impact:
  - Custom roles and fine-grained permissions cannot fully constrain core CRM CRUD paths
  - Authorization is pushed into service-level staff filtering rather than explicit endpoint policy
  - README claim of permission gates on all endpoints is currently inaccurate
- Recommendation:
  - Apply `PermissionsGuard` + `@RequirePermissions()` on mutating and sensitive read endpoints for core modules
  - Keep service-level access checks as defense-in-depth, not the primary policy layer
- Effort: 8-20h
